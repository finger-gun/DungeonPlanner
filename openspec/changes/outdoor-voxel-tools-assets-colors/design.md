## Context

DungeonPlanner outdoor authoring currently combines stepped terrain height data with texture-driven ground rendering. During exploration, we confirmed the new KayKit pack introduces a much larger terrain-oriented set (hill/cliff families and full Color1-Color8 variants) than what is currently ingested and registered. The goal is to upgrade outdoor terrain to a performant voxel-style workflow that uses these assets directly while preserving existing indoor architecture and editing speed.

Current constraints:
- Outdoor state and tools are centralized in `src/store/useDungeonStore.ts`.
- Outdoor rendering is currently generated by `OutdoorGround.tsx` from height + texture maps.
- Indoor rendering and batching paths (`DungeonRoom`, `BatchedTileEntries`) are stable and must not regress.
- Serialization compatibility must be maintained for existing saved maps.

## Goals / Non-Goals

**Goals:**
- Deliver outdoor voxel-style block elevation editing with responsive sculpting.
- Integrate expanded terrain assets and color variants (including missing hill/cliff sets and tree/rock/bush color families).
- Add user-facing terrain color palette controls and apply them consistently across elevate/surroundings/texture workflows.
- Keep default base ground behavior intuitive (green grass).
- Maintain predictable performance through chunk-local updates, dirty-region rebuilds, and exposed-face-only geometry output.
- Preserve indoor mode behavior and rendering architecture unchanged.

**Non-Goals:**
- Re-architecting indoor map data or renderer to voxels.
- Introducing full volumetric cave/overhang editing in this change.
- Replacing all content-pack systems beyond outdoor terrain/palette integration scope.

## Decisions

### 1) Use column-voxel terrain semantics instead of full freeform volume
- **Decision:** Represent outdoor elevation as voxel-aligned height columns on the existing XY grid semantics (cell + integer level stack), not unconstrained 3D volume editing.
- **Rationale:** Preserves current tactical grid behavior, simplifies pathing/placement compatibility, and keeps tool interaction familiar.
- **Alternative considered:** Full 3D volumetric voxels with caves/overhangs. Rejected for this phase due to larger UX, data, and interaction complexity.

### 2) Use chunked terrain processing with exposed-face generation
- **Decision:** Partition outdoor terrain into chunks and rebuild only dirty chunks and immediate neighbors after edits; emit only exposed voxel faces/surfaces.
- **Rationale:** Prevents full-scene rebuild costs and keeps sculpting responsive.
- **Alternative considered:** Full terrain remesh on every stroke. Rejected for expected performance regression on larger maps.

### 3) Keep indoor renderer and behavior untouched
- **Decision:** Isolate voxel terrain logic to outdoor mode; do not alter `DungeonRoom` indoor rendering/data semantics.
- **Rationale:** Limits regression risk and supports incremental delivery.
- **Alternative considered:** Shared generalized voxel renderer for indoor/outdoor. Rejected as unnecessary coupling for this scope.

### 4) Add palette-driven outdoor terrain styling
- **Decision:** Add global outdoor terrain palette selection (Color1-Color8) with optional tool-level usage by surroundings/terrain/texture workflows; default maps start with green grass baseline.
- **Rationale:** Delivers requested artistic control while keeping defaults simple and usable.
- **Alternative considered:** Hard-code one color family. Rejected because it does not satisfy flexibility goals.

### 5) Expand content-pack ingestion to complete terrain-ready set
- **Decision:** Ingest and register missing terrain-focused assets and color variants from the provided pack, including hill/cliff pieces and additional natural prop variants used by surroundings tools.
- **Rationale:** Voxel terrain rendering and palette workflows require complete asset coverage.
- **Alternative considered:** Keep existing curated subset only. Rejected due to visible feature gaps and incomplete terrain tooling outcomes.

### 6) Preserve compatibility through deterministic migration
- **Decision:** Add versioned outdoor migration from existing terrain representation to voxel-aligned column data with deterministic defaults for palette/base-ground behavior.
- **Rationale:** Existing saves must remain loadable without manual intervention.
- **Alternative considered:** Breaking format reset for outdoor maps. Rejected due to unacceptable user impact.

## Risks / Trade-offs

- **[Risk]** Chunk rebuild logic can create seam or neighbor desync bugs at chunk boundaries. → **Mitigation:** Always mark adjacent chunks dirty when edge cells change; add boundary-focused tests.
- **[Risk]** Asset count growth (colors + terrain families) increases memory and load-time pressure. → **Mitigation:** Lazy-load where possible, keep curated terrain sets per tool mode, and avoid loading unused color families eagerly.
- **[Risk]** Migration from legacy outdoor maps could produce unexpected visual differences. → **Mitigation:** deterministic conversion rules, green-grass fallback, and explicit backward-compat scenarios in specs/tests.
- **[Trade-off]** Column-voxel approach defers caves/overhangs. → **Mitigation:** Keep extensible data boundaries so volumetric expansion can be added later.

## Migration Plan

1. Introduce new outdoor terrain data fields and version markers while retaining legacy read support.
2. On load, convert legacy outdoor height/texture data to voxel-aligned terrain columns and palette defaults.
3. Persist the new outdoor format on next save, preserving deterministic round-trip behavior.
4. Keep fallback paths for files missing new palette fields (default to green grass baseline and default palette variant).
5. Provide rollback safety by keeping migration isolated to outdoor mode parsing and preserving unchanged indoor schemas.

## Open Questions

- Should per-tool palette override be included in v1, or should initial release ship with global palette only?
- Should texture brush expose full Color1-Color8 immediately, or only a curated subset aligned to base terrain styles?
